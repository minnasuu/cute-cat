## create-craft 输入解析修复

`create-craft.ts` 修复了上游步骤传递 JSON 对象时无法识别的问题。原代码只支持从 `_params` 字段中提取 JSON 字符串再 `JSON.parse`，但工作流引擎通过 `Object.assign(merged, prevData)` 传递上游结果时，`ctx.input` 已经是一个包含 craft 字段（name、htmlCode 等）的对象，而非 JSON 字符串。重写后支持三种输入格式：1) 直接 JSON 字符串；2) 对象中含 JSON 字符串字段；3) 对象本身就是 craft 数据。新增 `isCraftLike` 判断、`extractCraftFields` 字段提取（排除 `_action`/`_params` 等元字段）、`parseJsonToItems` JSON 解析三个辅助函数。

## manage-workflow 技能 — 增删改重构

重写了 `manage-workflow.ts`，从仅支持创建变为支持三种操作（create / update / delete）：

### 参数变更（skills.ts）
- 新增 `action`（select：🆕 新增工作流 / ✏️ 修改工作流 / 🗑️ 删除工作流，必填）
- 新增 `workflowId`（select，从 API `/api/workflows/team/:teamId` 异步加载当前团队工作流列表，修改/删除时必选）
- 保留 `prompt`（textarea，`valueSource: 'upstream'`）
- 删除 `teamId`（改为自动从 URL 或用户团队列表获取，只操作自己的团队）

### 三种操作实现
1. **create**：获取团队猫猫信息 → 调用 AI 生成工作流配置 → `POST /api/workflows/team/:teamId`
2. **update**：先 `GET /api/workflows/:id` 获取现有工作流 → 拼接修改需求+现有信息+猫猫信息 → AI 重新生成 → `PUT /api/workflows/:id`
3. **delete**：获取工作流名称 → `DELETE /api/workflows/:id`

### StepParam 类型扩展（types.ts）
新增三个可选字段支持 select 参数动态加载选项：
- `asyncOptionsFrom`：API 路径（如 `/api/workflows/team/:teamId`，`:teamId` 运行时从 URL 替换）
- `asyncOptionsValueKey`：选项 value 取值字段，默认 `'id'`
- `asyncOptionsLabelKey`：选项 label 取值字段，默认 `'name'`

### StepConfigPanel UI 支持（StepConfigPanel.tsx）
新增 `useAsyncOptions` Hook，在 `ParamInput` 的 `select` 分支中：
- 检测 `param.asyncOptionsFrom` 是否存在
- 存在时调用 API 加载选项，显示「加载中...」状态
- 自动替换路径中的 `:teamId` 占位符
- 异步选项和静态 `options` 互斥使用

AI 生成结构化数据（如 MECE 拆解、SCAMPER 创意、六顶思考帽等）时，因 `maxTokens` 默认值 2048 过小导致 JSON 输出被截断、解析失败。

### 变更
- **`backend/routes/dify.js`**：
  - `callQwen` 默认 `maxTokens` 从 2048 → 4096
  - `TASK_MAX_TOKENS` 映射表扩展，所有结构化输出类 skill（workflow-gen, mece-analysis, scamper-creative, six-hats, generate-outline, content-review, recruit-cat, team-review, cat-training）设为 8192
  - 其余未列出的 taskId 默认 4096
- **`backend/workflow-executor.js`**：`callQwen`、`callGemini`、`callAI` 三个函数默认 `maxTokens` 均从 2048 → 4096
- **`backend/prisma/schema.prisma`**：`TeamCat.maxTokens` 默认值从 2048 → 4096

### 变更文件

- **`WorkflowEditorPage.tsx`**：
  - 初始步骤 / `addStep` 自动生成 `stepId`
  - 从后端加载步骤时 `ensureStepIds()` 补全
  - AI 生成步骤也 `ensureStepIds()`
  - `handleConnect` 存 `stepId` 而非 `agentId`
  - `removeStep` 时清理引用被删除步骤的 `inputFrom`
- **`EdgePopover.tsx`**：选择和高亮用 `stepId` 匹配
- **`EdgeLines.tsx`**：使用 `resolveInputFromIndex()` 解析连线

## 工作流 DAG 分叉布局与并行执行

### 问题
当工作流出现分叉（一个步骤的输出被多个下游步骤使用），画布中所有节点重叠在一条直线上，无法展示分叉结构；后端执行也是串行顺序执行，没有利用分叉的并行性。

### 方案
1. **前端画布 DAG 布局**：分析 `inputFrom` 依赖关系构建 DAG，按拓扑层级（depth）排列节点，同层分叉节点水平展开
2. **后端 DAG 并行执行**：构建同样的 DAG 拓扑，同层（互不依赖）的步骤用 `Promise.all` 并行执行

### 变更文件
- **`canvas-utils.ts`**：
  - `autoLayout` 函数重载：支持传入 `WorkflowStep[]`（DAG 感知布局）或 `number`（fallback 简单瀑布布局）
  - DAG 布局算法：解析 `inputFrom` → 计算拓扑层级 → 同层节点水平展开（`NODE_GAP_X = 40`）
  - 新增 `autoLayoutSimple` 简单布局函数
  - 新增 `NODE_GAP_X` 常量
- **`EdgeLines.tsx`**：
  - 开始节点连线：从只连第一个步骤改为连接所有"根"步骤（`parentIndex === -1`）
  - 开始节点连线索引使用 `-(i+2)` 保证唯一性
  - 删除按钮判断从 `index > 0` 改为 `index >= 1`
- **`WorkflowEditorPage.tsx`**：
  - `autoLayout` 调用从传 `steps.length` 改为传完整 `steps` 数组
  - `handleEdgeClick` 判断从 `index <= 0` 改为 `index < 1`
  - Minimap 新增 `steps` prop
- **`Minimap.tsx`**：
  - 新增 `steps` prop 和 `WorkflowStep` 类型导入
  - 连线绘制从简单顺序连线改为基于 `resolveInputFromIndex` 的 DAG 拓扑连线
- **`workflow-executor.js`**：
  - `executeWorkflow` 重写为 DAG 分层并行执行
  - 解析 `inputFrom`（优先 stepId、fallback agentId）构建 `parentIndex`
  - 按拓扑层级分组，同层步骤 `Promise.all` 并行执行
  - 每个步骤获取自己上游步骤的输出（而非全局累积的 prevResults）
  - 执行结果按步骤索引排序输出

## create-craft 技能 JSON 解析容错修复

### 问题
上游步骤（AI 生成 craft 数据）返回的 JSON 字符串中，`htmlCode` 字段包含的 HTML 内有未转义的双引号（如 `charset="utf-8"`），导致 JSON 解析失败报错。

### 修复（两个文件）

#### `frontend/src/skills/admins/create-craft.ts`
1. **提取优先级调整**：对象类型输入优先检查 `isCraftLike`（情况 3）→ 再尝试从字段提取 JSON 字符串（情况 2），避免把已有的 craft 对象做不必要的 JSON 字符串化→解析循环
2. **新增 `extractJsonFromText`**：支持从 markdown 代码块包裹的文本中提取 JSON，支持去除前后多余文本
3. **新增 `fixHtmlCodeQuotes`**：修复 `htmlCode` 字段值中嵌入的未转义双引号——找到 `"htmlCode": "` 后定位值的真正结尾，对中间的裸双引号转义
4. **新增 `extractCraftFromBrokenJson`**：JSON 解析完全失败时的最终 fallback，用正则从破碎 JSON 中提取 `name`、`htmlCode` 等关键字段
5. **错误信息增强**：`catch` 块输出详细错误信息和原始输入到 console

#### `backend/workflow-executor.js`
- `executeStep` 新增 `create-craft` / `view-crafts` 技能分支
- 不再走 AI 通道（避免将 craft 数据当文本处理），直接从 `merged` 提取 craft 字段或解析 JSON 字符串后原样输出
- 支持从上游直接传递的对象字段（`merged.name && merged.htmlCode`）和 JSON 字符串两种来源

## 工作流 DAG 分层并行执行改造

### 需求
工作流有分叉时（步骤 `inputFrom` 相同），同层步骤应同时并行执行，从上到下分层展示。

### 改造（`frontend/src/pages/TeamDetailPage/index.tsx`）

#### 执行引擎
- 从单步线性推进（`runningStepIndex: number`）改为 DAG 分层并行执行（`runningStepIndices: Set<number>`）
- `handleConfirmRun` 中构建 DAG 拓扑：解析 `inputFrom` → 计算 `parentIndex` → 按 `depth` 分层 → 存入 `dagLayers` / `dagParentIndex` 状态
- 执行 `useEffect` 中同层所有步骤并行执行（`Promise.all`），层级结束后自动启动下一层可执行步骤
- 上游数据合并改为按 `dagParentIndex` 精确取直接上游结果，而非遍历所有前序步骤
- 使用 `completedStepsRef` 解决 `Promise.all` 回调中的闭包问题

#### UI 渲染
- 执行态/历史查看态的 pipeline 从线性 `.map(step)` 改为按 DAG 层级分组渲染
- 同层多步骤水平并排（`display: flex`、`gap: 12`）
- 层间保留箭头连接
- Footer 进度显示更新：显示已完成数 + 当前并行数

#### 状态变量变更
- `runningStepIndex: number` → `runningStepIndices: Set<number>`
- 新增 `dagLayers: number[][]`、`dagParentIndex: number[]`、`completedStepsRef`
- 移除 `timerRef`（不再需要）
